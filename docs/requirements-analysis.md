# 第 2 章 需求分析

## 2.1 需求概述

### 2.1.1 项目背景

国有企业、事业单位和大型企业通常拥有大量固定资产，涵盖办公设备、电子设备、运输设备、生产设备和房屋建筑物等类别，资产数量从数百到数万不等，总价值往往达到千万元甚至上亿元级别。这类资产的日常管理涉及购置登记、领用分配、跨部门调拨、定期维修、折旧计提、报废处置等多个环节，管理链条长、参与角色多、合规要求高。

当前绝大多数国企仍以 Excel 表格或手工台账作为固定资产管理工具，存在以下突出问题：

1. **资产台账分散**。各部门各自维护自己的资产清单，资产编号不统一、分类口径不一致，总账与明细账难以核对，资产总数和总价值无法实时汇总。
2. **流转流程不透明**。资产领用靠纸质申请单，调拨靠电话或邮件通知，维修无正式记录，报废流程随意。管理人员无法追溯「这个资产现在在谁手里、经历了哪些变更」。
3. **盘点效率低**。年度盘点依赖人工逐项核对 Excel 清单与实物，部门多、地点分散的国企往往需要数周才能完成一次盘点，盘点结果整理成报告又需额外时间，盘盈盘亏难以追溯原因。
4. **折旧统计困难**。不同资产分类有不同折旧年限（办公设备 5 年、电子设备 4 年、车辆 8 年、房屋 20 年），人工计算工作量大且容易出错，财务部门月末结账时常因资产折旧数据延迟而受影响。
5. **操作记录缺乏审计留痕**。Excel 台账无操作日志，谁在什么时候新增了什么资产、修改了什么字段、删除了什么记录，无从查证。审计检查时只能依赖纸质单据，效率低且有丢失风险。
6. **缺少扩展能力**。存量系统多为单机版或简单的 Web 应用，无法提供 AI 辅助分析、网页端财务数据查看等现代化管理手段，也未预留二维码扫码、RFID 批量识别等后续扩展接口。

基于以上背景，本团队决定开发一套「国企固定资产全生命周期管理系统」，以数字化手段覆盖资产从入库到报废的全过程管理。

### 2.1.2 目标用户

本系统的目标用户按职责角色分为以下四类：

| 用户角色 | 所属部门 | 核心职责 |
|---|---|---|
| 系统管理员 | 信息中心 | 系统配置、用户管理、权限分配、数据维护 |
| 资产管理员 | 资产管理部 | 资产台账维护、生命周期单据操作、盘点执行、资产统计 |
| 财务人员 | 财务部 | 折旧计提核对、资产原值与净值审核、财务数据导出 |
| 审计人员 | 审计部/纪委 | 资产台账查询、操作日志审计、报表审查 |

其中，资产管理员是本系统最核心、最活跃的用户群体，系统超过 70% 的日常操作由该角色完成。

### 2.1.3 要解决的核心问题

本系统聚焦解决以下五个核心问题：

1. **资产家底不清**：通过统一的资产台账，让企业实时掌握资产总数、原值总和、累计折旧、净值总和等核心指标，做到「资产数据一个库、资产台账一张表」。
2. **流转无痕**：通过入库单、领用单、调拨单、维修单、报废单等生命周期单据，为每一次资产状态变化建立正式的业务记录，全程留痕、可追溯、可审计。
3. **盘点低效**：通过盘点任务管理、扫码盘点、差异自动比对、盘点报告生成，将一次全企业盘点从数周缩短到数天。
4. **折旧不准**：通过系统自动按平均年限法计算月度折旧，逐步积累真实折旧记录，替代人工 Excel 计算。
5. **系统封闭**：预留二维码/RFID 标签接口、财务系统对接接口、AI 分析接口，使系统具备持续扩展能力。

### 2.1.4 系统建设目标

本系统的最终建设目标为：

- **近期目标（第一阶段 + 第二阶段）**：搭建系统骨架，完成登录鉴权、首页驾驶舱、资产台账管理，以及入库、领用、调拨、维修、报废五个生命周期单据模块，实现资产台账数字化和状态流转规范化。
- **中期目标（第三阶段至第六阶段）**：引入审批流、盘点管理、折旧报表与财务对接、用户管理与权限控制，使系统具备完整的企业级资产管理能力。
- **远期目标（第七阶段至第八阶段）**：接入 AI 资产照片分类与故障预测、RFID 批量盘点、二维码标签打印、Docker 容器化部署，形成可对外演示和交付的完整产品。

### 2.1.6 AI 辅助生成与小组讨论修改说明

本需求分析文档的初稿由 AI 辅助生成，生成过程如下：

1. **AI 初稿生成**：项目组成员将项目实际代码（后端 Controller、Service、Entity 和前端页面、路由配置）的关键信息整理后，输入 AI 工具，要求 AI「根据实际项目的代码结构和已完成功能，生成需求分析文档」。

2. **小组第一次评审**：小组 6 名成员在第一次讨论会上逐段审阅 AI 初稿，发现以下问题并进行了修改：
   - AI 初稿中的「项目背景」过于通用（如提到「部分企业还在使用纸质管理」），结合团队调研的本校后勤集团和本地一家国企的实际管理情况，补充了具体的场景描述和痛点细节。
   - AI 初稿将「资产折旧计算」列在第二阶段功能范围中，但团队讨论后认为第二阶段聚焦单据流，折旧报表应归入第五阶段。
   - AI 初稿的用例描述中缺失了「异常流程」部分，小组统一补充了每个用例的异常场景。
   - AI 初稿中的「用户角色」只列出了系统管理员和资产管理员两个角色，小组讨论后补充了财务人员和审计人员角色，因为国企资产管理必须包含财务对账和审计追溯场景。
   - AI 初稿中的部分表述采用了英文技术术语（如「Workflow」「Approval」），小组统一改为中文表达。

3. **第二次评审**：小组将修改后的文档提交指导教师审阅，根据意见调整了功能范围的分类粒度，并增加了 2.1.3 核心问题描述和 2.1.6 AI 辅助说明章节。

4. **最终定稿**：经小组全体成员确认后定稿。

### UC-01：用户登录与权限访问

| 项目 | 内容 |
|---|---|
| **用例编号** | UC-01 |
| **用例名称** | 用户登录与权限访问 |
| **参与者** | 系统管理员、资产管理员、财务人员、审计人员 |
| **前置条件** | 1. 系统已部署并启动；2. 数据库已初始化，用户数据已导入；3. 用户拥有有效的账号密码 |
| **主流程** | 1. 用户打开系统首页，自动跳转至登录页面；2. 用户在登录表单中输入用户名和密码；3. 系统校验用户名是否存在、密码是否正确（BCrypt 加密比对）；4. 校验通过后，系统生成 JWT Token 返回给前端，Token 中包含用户 ID、用户名、角色列表等信息；5. 前端将 Token 存入 localStorage；6. 系统根据用户角色加载对应的菜单权限，跳转至首页驾驶舱；7. 用户后续的所有 API 请求均在请求头中自动携带 `Authorization: Bearer <token>`；8. 后端拦截器对每次请求解析 Token，通过后将用户信息写入 ThreadLocal（UserContext）供后续业务使用 |
| **异常流程** | **A. 用户名或密码错误**：系统返回 400 错误，前端提示「用户名或密码错误」，登录表单不清空，用户可以重新输入；**B. Token 过期或无效**：后端返回 HTTP 401，Axios 响应拦截器捕获到 401 后清除 localStorage 中的 Token 和用户信息，弹出「登录已失效，请重新登录」提示，自动跳转至登录页；**C. 无 Token 访问受保护资源**：后端拦截器捕获到请求头中无 Authorization 字段，直接返回 HTTP 401，前端跳转登录页；**D. 连续输错密码**：当前版本未做登录失败次数限制，后续版本可引入验证码和账户锁定机制 |
| **后置条件** | 1. 用户成功登录，JWT Token 有效；2. UserContext 中保存了当前登录用户的信息；3. 前端路由至目标页面 |

### UC-02：资产台账管理

| 项目 | 内容 |
|---|---|
| **用例编号** | UC-02 |
| **用例名称** | 资产台账管理 |
| **参与者** | 资产管理员（主要）、系统管理员 |
| **前置条件** | 1. 用户已登录并拥有资产台账管理权限；2. 数据库中已存在资产分类数据和至少一条资产数据 |
| **主流程** | 1. 用户点击左侧导航「资产管理 → 资产台账」，进入资产台账列表页面；2. 页面自动加载资产分页数据，默认每页 10 条，展示字段包括：资产编号、资产名称、分类、品牌、规格型号、部门、使用人、原值、净值、状态、购置日期；3. 用户可以通过查询表单按资产编号、资产名称、分类、部门、使用人、存放地点、状态、购置日期范围等条件筛选资产，点击「查询」按钮刷新列表，点击「重置」按钮清空所有查询条件；4. 用户点击「新增资产」按钮，打开新增弹窗，填写资产名称、分类、品牌、规格型号、购置日期、原值、使用年限、残值率、所属部门、使用人、存放地点等信息，资产编号由后端自动生成（无需填写），累计折旧和净值由后端自动计算，点击「确认」提交；5. 用户点击操作列的「编辑」按钮，打开编辑弹窗回显当前数据，修改需要变更的字段后保存，系统自动重新计算折旧和净值；6. 用户在编辑弹窗中可以选择修改资产状态（IDLE / IN_USE / REPAIRING / WAITING_SCRAP / SCRAPPED 等），状态变更时自动写入 `asset_operation_log`；7. 用户点击「删除」按钮，系统弹出二次确认对话框，确认后执行逻辑删除（`deleted = 1`），该资产不再出现在列表中；8. 用户点击操作列的「查看」按钮，跳转至资产详情页面，展示资产的完整信息（基础信息、价值信息、使用信息、状态信息、标签信息、备注）；9. 所有新增、编辑、删除操作完成后主动刷新表格 |
| **异常流程** | **A. 资产分类不存在**：用户提交新增时指定的 categoryId 在数据库中无对应记录，系统返回 400「资产分类不存在」；**B. 资产不存在**：编辑或删除时指定的 ID 在数据库中无对应记录或已被逻辑删除，系统返回 404「资产不存在」；**C. 必填字段缺失**：前端表单校验阻止提交，如资产名称、分类、原值等字段为空时 El-Form 校验不通过，提示对应错误信息；**D. 操作日志写入失败**：系统使用事务回滚，业务操作不生效 |
| **后置条件** | 1. 新增：资产表中新增一条记录，状态默认为 IDLE，自动生成资产编号，写入操作日志；2. 编辑：资产表中对应记录被更新，写入操作日志；3. 删除：资产表对应记录的 deleted 字段标记为 1，写入操作日志；4. 查询：返回符合条件的资产分页数据 |

### UC-03：生命周期单据流转

| 项目 | 内容 |
|---|---|
| **用例编号** | UC-03 |
| **用例名称** | 生命周期单据流转 |
| **参与者** | 资产管理员 |
| **前置条件** | 1. 用户已登录并拥有生命周期管理权限；2. 数据库中存在至少一条 IDLE 状态的资产（用于领用）和至少一条 IN_USE 或 IDLE 状态的资产（用于调拨/维修） |
| **主流程（领用场景）** | 1. 用户点击左侧导航「生命周期 → 资产领用」，进入领用管理页面；2. 用户点击「新增领用」按钮，打开新增弹窗；3. 用户在资产选择器中选择一条 IDLE 状态的资产（系统自动过滤，仅显示 IDLE 资产）；4. 用户填写领用人姓名、领用部门、领用日期和选填的使用用途；5. 点击「确认领用」提交；6. 系统校验资产状态为 IDLE，执行领用操作：创建领用单（单据编号自动生成为 RE+年月日+流水号），资产 status 更新为 IN_USE，asset.department 更新为领用部门，asset.keeper 更新为领用人，写入 `asset_operation_log`；7. 弹窗关闭，列表自动刷新，显示新创建的领用单据 |
| **主流程（维修 → 维修完成场景）** | 1. 用户进入维修管理页面，点击「新增维修」；2. 选择资产（IDLE 或 IN_USE），填写故障描述、维修厂商等信息，提交后资产状态变为 REPAIRING，维修单状态为 DRAFT；3. 维修完工后，用户在维修列表中找到该 DRAFT 状态的单据，点击「完成」按钮；4. 页面弹出完成维修弹窗，用户选择维修结果（REPAIRED 修复完成 / SCRAP_SUGGESTED 建议报废），填写维修厂商、实际费用、完成日期等信息；5. 提交后系统校验维修单为 DRAFT 状态，更新资产状态：REPAIRED → IN_USE，SCRAP_SUGGESTED → WAITING_SCRAP；6. 维修单状态变更为 COMPLETED，写入操作日志 |
| **主流程（报废场景）** | 1. 用户进入报废管理页面，点击「新增报废」；2. 选择资产（允许 IDLE / IN_USE / REPAIRING / WAITING_SCRAP 状态），填写报废原因、报废日期、处置方式、残值等信息；3. 提交后系统校验资产允许报废，创建报废单，资产状态变为 SCRAPPED，写入操作日志 |
| **异常流程** | **A. 领用非 IDLE 资产**：系统返回 400「仅闲置资产可领用，当前状态:XXX」；**B. 调拨/维修/报废已报废（SCRAPPED）资产**：系统返回 400「当前状态不允许XX」；**C. 维修单重复完成**：已完成（COMPLETED）的维修单再次点击完成时，系统返回 400「维修单状态不是草稿」；**D. 参数缺失**：如领用场景的领用人、领用部门为空，前端表单校验阻止提交 |
| **后置条件** | 1. 创建单据后，对应生命周期单据表中新增一条记录，资产表状态按照流转规则更新；2. 所有涉及资产状态变更的操作均写入 `asset_operation_log` 表；3. 列表刷新 |

| **参与者** | 资产管理员、系统管理员 |
| **前置条件** | 1. 用户已登录；2. 数据库中已有有效资产记录；3. 审批流（第三阶段）已完成开发（盘点流程需要审批支持） |
| **主流程** | 1. 资产管理员创建盘点任务，填写任务名称、选择盘点范围（按部门 / 按地点 / 全部资产）；2. 系统根据盘点范围自动生成盘点清单，列出需要盘点的资产及其应在地点和应在保管人；3. 资产管理员打印盘点清单或导出至移动设备，到现场逐项核对资产实物；4. 现场核对方式支持：二维码扫码（扫描资产上的二维码标签自动定位到盘点项）、手动勾选正常、手动录入异常；5. 盘点结果包括：正常（账实相符）、盘盈（实物存在但系统无记录）、盘亏（系统有记录但实物缺失）、位置异常（实物不在应在地点）；6. 盘点完成后生成盘点报告，统计正常数量、盘盈数量、盘亏数量、异常数量；7. 盘盈资产可一键新增至资产台账，盘亏/异常资产可标记为「盘点异常」状态（INVENTORY_ABNORMAL）并触发审批流程 |
| **异常流程** | **A. 盘点进行中系统故障**：盘点数据支持暂存，任务支持暂停和恢复，不要求一次性完成；**B. 资产被重复盘点**：同一资产在同一任务中只能被扫码一次，重复扫码提示「已盘点」；**C. 盘点清单与实物严重不符**：允许资产管理员在盘点过程中将异常资产标记为「暂不处理」，留待后续专项核查 |
| **后置条件** | 1. 盘点任务状态变更为 COMPLETED；2. 盘点报告生成，数据持久化到 inventory_task 和 inventory_record 表；3. 异常资产状态标记更新 |

> **说明**：UC-04 为设计用例，当前系统尚未实现。数据库（inventory_task、inventory_record 表）已在第一阶段预留表结构。完整功能在第四阶段开发。

### UC-05：折旧报表与财务同步（设计用例）

| 项目 | 内容 |

| **参与者** | 财务人员 |

## 2.3 AI 辅助与人工修订记录

### 2.3.1 辅助过程

本需求分析文档的生成采用了「AI 驱动 + 小组人工审核」的协作方式，具体过程如下：

**第一步：AI 初稿生成**

项目组成员将以下材料输入 AI 工具：
1. 项目 README.md（包含完整的已完成功能清单、技术栈、目录结构、验收结果）
2. 项目 docs/sdd/ 目录下的阶段计划文档（00-project-scope.md）、领域模型文档（01-domain-model.md）
3. 已实现的后端 Controller 和 Service 关键代码片段的接口路径和业务逻辑说明
4. 前端路由配置和页面清单

AI 基于上述材料生成了初始版本的需求分析文档，覆盖了项目背景、目标用户、功能范围和 5 个核心用例的初稿框架。

**第二步：小组第一次讨论修改（2026 年 7 月 5 日）**

| 问题 | AI 初稿内容 | 小组修改 |
|---|---|---|
| 项目背景过于通用 | 「部分企业还在使用纸质管理表格」——表述模糊，缺少具体场景 | 替换为具体的国企管理痛点描述（台账分散、流程不透明、盘点效率低、折旧困难、无审计留痕、缺少扩展能力） |
| 用户角色遗漏 | 仅列出「系统管理员」和「资产管理员」两个角色 | 补充「财务人员」（负责折旧核对和财务数据导出）和「审计人员」（负责台账查询和操作日志审计）两个角色 |
| 用例分类偏差 | 将「折旧计算」归入第二阶段功能范围 | 第二阶段聚焦单据流，折旧报表归入第五阶段 |
| 缺乏异常流程 | 每个用例只有主流程描述，无异常处理 | 为每个用例补充了 2-4 个异常场景，包括业务异常和系统异常 |
| 技术术语未中文化 | 使用了「Workflow」「Approval」「Dashboard」等英文术语 | 统一替换为「工作流」「审批」「驾驶舱」等中文表达 |
| 前置条件不完整 | 仅描述简单的「用户已登录」 | 补充了数据库数据前提（如必须存在 IDLE 资产才能进行领用测试） |

**第三步：指导教师审阅（2026 年 7 月 6 日）**

指导教师审阅后提出以下修改意见：

1. 功能范围应分阶段列出，便于阅读者了解系统整体规划和当前进度。
2. 用例格式应增加「后置条件」字段，描述操作完成后系统数据的变化。
3. 设计用例（UC-04、UC-05）应明确标注「设计用例」标识，避免与已实现的用例混淆。
4. 增加 AI 辅助说明章节，体现团队在 AI 辅助下的独立思考和判断能力。

**第四步：最终定稿**

小组根据指导教师意见逐项修改完成后，经全体 6 名成员确认定稿。

### 2.3.3 局限性说明

1. AI 生成的内容依赖于输入材料的完整性和准确性。如果输入材料中缺少某些功能的说明（如「当前版本未做登录失败次数限制」），AI 初稿中也会缺失这些细节，需要小组成员根据实际开发经验补充。
2. AI 在描述用例异常流程时倾向于生成「通用的异常场景」（如「网络错误」「系统异常」），而不是「业务领域特有的异常场景」（如「已报废资产不能再次报废」）。后者需要小组成员根据实际业务规则逐一补充。
3. AI 对系统的「阶段规划」理解不充分——AI 倾向于将所有功能列在一起，而实际开发需要分阶段逐步交付。小组在审阅时需要根据当前实际进度调整功能的阶段归属。
4. 本需求分析文档中的设计用例（UC-04、UC-05）尚未经过代码实现验证，后续开发时可能因技术或业务原因产生调整，届时需同步更新本文档。
