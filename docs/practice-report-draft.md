# 国企固定资产全生命周期管理系统实践报告（初稿）

---

## 第一章 项目简介

### 1.1 项目背景

国有企业、事业单位和大型企业通常拥有大量固定资产，涵盖办公设备、电子设备、运输设备、生产设备和房屋建筑物等类别，资产数量从数百到数万不等，总价值往往达到千万元甚至上亿元级别。这类资产的日常管理涉及购置登记、领用分配、跨部门调拨、定期维修、折旧计提、报废处置等多个环节，管理链条长、参与角色多、合规要求高。

当前绝大多数国企仍以 Excel 表格或手工台账作为固定资产管理工具，存在以下突出问题：

1. **资产台账分散**。各部门各自维护资产清单，编号不统一、分类口径不一致，总账与明细账难以核对。
2. **流转流程不透明**。领用靠纸质申请单，调拨靠电话通知，维修无正式记录，报废流程随意。
3. **盘点效率低**。年度盘点依赖人工逐项核对，部门多、地点分散时往往需要数周。
4. **折旧统计困难**。不同分类有不同折旧年限，人工计算工作量大且容易出错。
5. **操作记录缺乏审计留痕**。Excel 台账无操作日志，谁在何时修改了什么无从查证。
6. **缺少扩展能力**。无法提供 AI 辅助分析、网页端财务数据查看等现代化管理手段，也未预留二维码扫码、RFID 等后续扩展接口。

基于以上背景，本项目开发一套「国企固定资产全生命周期管理系统」，以数字化手段覆盖资产从入库到报废的全过程管理。

### 1.2 项目目标

本系统聚焦解决以下五个核心问题：

1. **资产家底不清**：通过统一资产台账，实时掌握资产总数、原值总和、累计折旧、净值总和。
2. **流转无痕**：通过入库单、领用单、调拨单、维修单、报废单等生命周期单据，全程留痕、可追溯、可审计。
3. **盘点低效**：通过盘点任务管理、差异自动比对，提升盘点效率（扫码盘点等实物能力作为后续扩展，不纳入本阶段）。
4. **折旧不准**：通过系统自动按平均年限法计算月度折旧，替代人工 Excel 计算。
5. **系统封闭**：AI 辅助分析模块可生成资产状态摘要和异常提示；网页端可查看财务数据与模拟同步记录；二维码/RFID 等实物能力作为后续扩展方向，不纳入本阶段范围。

### 1.5 开发方法：AI IDE 辅助开发

本项目全程借助 AI IDE（Trae）辅助开发，采用「AI 驱动 + 人工审核」的协作模式：

1. **需求分析阶段**：将项目背景和调研材料输入 AI，生成需求分析初稿，小组成员讨论后补充异常流程、修正角色定义、中文化术语。
2. **系统设计阶段**：基于需求文档，由 AI 辅助生成数据库表结构、接口设计、前端路由设计，人工审核后确认。
3. **编码实现阶段**：遵循 SDD（Spec-Driven Development，规格驱动开发）流程，按「数据库 → Entity → Mapper → DTO/VO → Service → Controller → 前端 API → 前端页面」的顺序逐层实现，AI 辅助生成代码骨架，人工补充业务逻辑。
4. **测试验收阶段**：每个阶段完成后执行 `mvn -DskipTests package` 和 `npm run build` 双构建验证，并使用 HTTP 接口测试和浏览器验收。

### 1.6 当前完成成果

截至报告撰写时，项目已完成第一阶段和第二阶段，主要成果包括：

- **后端**：14 张数据表（含 5 张生命周期单据表），涵盖认证、资产、生命周期、仪表盘、财务、盘点等模块的完整后端服务。
- **前端**：登录页、首页驾驶舱、资产台账、资产详情、资产分类、5 个生命周期单据页面，以及资产选择器、单据详情弹窗等公共组件。
- **接口**：17 个生命周期接口 + 资产管理接口 + 鉴权接口 + 仪表盘接口，统一采用 `Result<T>` 返回格式。
- **验收**：第一阶段和第二阶段全部验收项通过，包括干净数据库复测（40/40）、双构建验证、非法状态拦截测试（7 项全部 400）。

#### UC-01：用户登录系统

| 项目 | 内容 |
|---|---|
| **用例编号** | UC-01 |
| **参与者** | 系统管理员、资产管理员、财务人员、审计人员 |
| **前置条件** | 1. 系统已部署并启动；2. 数据库已初始化，用户数据已导入；3. 用户拥有有效账号密码 |
| **主流程** | 1. 用户打开系统首页，自动跳转至登录页；2. 输入用户名和密码；3. 系统校验用户名是否存在、密码是否正确（BCrypt 加密比对）；4. 校验通过后生成 JWT Token 返回前端，Token 中含用户 ID、用户名、角色列表；5. 前端将 Token 存入 localStorage；6. 根据角色加载菜单权限，跳转至首页驾驶舱；7. 后续所有 API 请求自动携带 Authorization: Bearer <token>；8. 后端拦截器解析 Token，写入 UserContext（ThreadLocal） |
| **异常流程** | A. 用户名或密码错误：返回 400，提示「用户名或密码错误」；B. Token 过期或无效：返回 HTTP 401，前端清除 Token 并跳转登录页；C. 无 Token 访问受保护资源：返回 401 跳转登录页；D. 连续输错密码：当前版本未做失败次数限制，后续可引入验证码 |
| **后置条件** | 1. 用户成功登录，JWT Token 有效；2. UserContext 保存当前登录用户信息；3. 前端路由至目标页面 |

### 2.4 资产状态流转规则

系统定义了 7 种资产状态，通过生命周期单据驱动状态流转：

| 状态码 | 说明 |
|---|---|
| IDLE | 闲置 |
| IN_USE | 使用中 |
| TRANSFERRING | 调拨中 |
| REPAIRING | 维修中 |
| WAITING_SCRAP | 待报废 |
| SCRAPPED | 已报废 |
| INVENTORY_ABNORMAL | 盘点异常 |

状态流转规则：

| 流程 | 操作 | 资产状态变化 |
|---|---|---|
| 入库 | 创建入库单 | 任意 → IDLE |
| 领用 | 创建领用单 | IDLE → IN_USE |
| 调拨 | 创建调拨单 | IDLE / IN_USE → IN_USE |
| 创建维修 | 创建维修单 | IDLE / IN_USE → REPAIRING |
| 维修完成 | repairResult = REPAIRED | REPAIRING → IN_USE |
| 维修完成 | repairResult = SCRAP_SUGGESTED | REPAIRING → WAITING_SCRAP |
| 报废 | 创建报废单 | IDLE/IN_USE/REPAIRING/WAITING_SCRAP → SCRAPPED |

所有状态变化自动写入 `asset_operation_log`，保证审计可追溯。

#### 3.3.5 核心业务规则

**资产编号规则**：`FA` + `yyyyMM` + `4位流水号`（如 FA2026070001），后端自动生成，每月流水号从 0001 开始。

**折旧计算规则**（平均年限法）：
- 年折旧额 = 原值 × (1 - 残值率) / 使用年限
- 月折旧额 = 年折旧额 / 12
- 累计折旧 = 已使用月份 × 月折旧额
- 资产净值 = 原值 - 累计折旧（净值不能小于残值）

**生命周期单据编号规则**：`前缀` + `yyyyMMdd` + `4位流水号`

| 单据类型 | 前缀 | 示例 |
|---|---|---|
| 入库单 | IN | IN202607050001 |
| 领用单 | RE | RE202607050001 |
| 调拨单 | TF | TF202607050001 |
| 维修单 | RP | RP202607050001 |
| 报废单 | SC | SC202607050001 |

跨 5 张表查询最大编号保证唯一性，每天流水号从 0001 开始。

**逻辑删除规则**：asset 表使用 `deleted` 字段标记，MyBatis-Plus 全局配置自动过滤 `deleted=0` 数据。

### 3.5 鉴权设计

#### 3.5.1 登录流程

```
用户输入账号密码
       |
       v
前端 POST /api/auth/login
       |
       v
后端 AuthService.login()
  ├── 查询 sys_user 表
  ├── BCrypt 校验密码
  ├── 查询用户角色
  ├── 生成 JWT Token（含 userId, username, roles）
  └── 返回 token + 用户信息
       |
       v
前端保存 token 到 localStorage → 跳转 /dashboard
```

#### 3.5.2 接口鉴权流程

```
前端请求 /api/**
       |
       v
Axios 请求拦截器注入 Authorization: Bearer <token>
       |
       v
AuthInterceptor.preHandle()
  ├── OPTIONS 请求放行
  ├── 检查 Authorization 头
  ├── 解析 JWT Token
  ├── 写入 UserContext（ThreadLocal）
  └── 放行到 Controller
       |
       v
Controller 处理业务
       |
       v
AuthInterceptor.afterCompletion() → 清除 UserContext
```

#### 3.5.3 401 处理与路由守卫

- **401 处理**：后端返回 HTTP 401 → Axios 响应拦截器捕获 → 清除 localStorage Token 和 Pinia 用户信息 → 提示「登录已失效」→ 跳转登录页。
- **路由守卫**：访问 `/login` 已登录则跳转 `/dashboard`；访问其他页面未登录则跳转 `/login`。

### 3.7 首页驾驶舱统计设计

驾驶舱从 asset 表实时统计，核心指标与 SQL 口径：

| 指标 | 来源 |
|---|---|
| 资产总数 | asset 表 COUNT（过滤 deleted=0） |
| 资产总原值 | SUM(original_value) |
| 累计折旧 | SUM(accumulated_depreciation) |
| 资产净值 | SUM(net_value) |
| 使用中/闲置/维修中/待报废数量 | 按 status 分组统计 |
| 分类分布 | LEFT JOIN asset_category GROUP BY |
| 部门排行 | GROUP BY department |
| 折旧趋势 | 第一阶段 mock 数据（后续接入真实数据） |

### 3.8 开发流程规范

本项目遵循 SDD（Spec-Driven Development，规格驱动开发）方式，每个新阶段必须执行：

```
Spec → Scope → Design → Tasks → Impl → Build → Accept → Review → Commit
```

**开发顺序**：数据库 → Entity → Mapper → DTO/VO → Service → Controller → 前端 API → 前端页面

**验收规范**：

| 检查项 | 要求 |
|---|---|
| 后端构建 | mvn -DskipTests package 通过 |
| 前端构建 | npm run build 通过 |
| 数据库迁移 | 迁移 SQL 执行成功 |
| 正常流程 | 所有正常路径可达 |
| 异常流程 | 所有非法路径返回正确错误 |
| 回归测试 | 不影响已有功能 |
| 操作日志 | 关键操作写入日志 |
