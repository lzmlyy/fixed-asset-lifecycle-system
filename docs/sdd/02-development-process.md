# 开发流程规范

## 1. 每个新阶段必须执行的步骤

```
Step 1: Spec    —— 创建需求规格文档（docs/sdd/phase-X-name.md）
Step 2: Scope   —— 明确本阶段做什么、不做什么
Step 3: Design  —— 设计数据库表、后端接口、前端页面、状态流转、异常规则
Step 4: Tasks   —— 将工作拆分为可执行的任务清单
Step 5: Impl    —— 按任务清单逐项实现代码
Step 6: Build   —— 后端 mvn -DskipTests package && 前端 npm run build
Step 7: Accept  —— 按验收用例逐项测试
Step 8: Review  —— 输出验收报告
Step 9: Commit  —— 通过后才能提交代码
```

## 2. 禁止事项

| 禁止 | 后果 |
|---|---|
| 未写 spec 直接开发 | 需求不清晰，反复改 |
| 未确认表结构直接建表 | 字段缺失或冗余，后期难改 |
| 未设计接口直接写前端 | 前后端无法对接 |
| 未跑验收直接说完成 | 遗留 Bug，导致不稳定 |
| 改坏第一阶段和第二阶段主链路 | 破坏整个系统可用性 |

## 3. 开发顺序规范

每个新功能模块的开发顺序：

1. **数据库** — 先设计表结构，再执行迁移 SQL
2. **Entity** — 先创建实体类
3. **Mapper** — 先写数据访问层
4. **DTO / VO** — 先定义请求和响应模型
5. **Service** — 再写业务逻辑
6. **Controller** — 再写接口层
7. **前端 API** — 再通后端接口
8. **前端页面** — 最后做页面展示

## 4. 验收规范

| 检查项 | 必须 |
|---|---|
| 后端构建 | mvn -DskipTests package 通过 |
| 前端构建 | npm run build 通过 |
| 数据库迁移 | 迁移 SQL 执行成功 |
| 正常流程 | 所有正常路径可达 |
| 异常流程 | 所有非法路径返回正确错误 |
| 回归测试 | 不影响已有功能 |
| 操作日志 | 关键操作写入日志 |
