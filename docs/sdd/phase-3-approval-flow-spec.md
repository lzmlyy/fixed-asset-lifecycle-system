# 第三阶段：审批流 · 需求规格说明

## 1. 背景

第二阶段已实现五个生命周期单据模块（入库、领用、调拨、维修、报废），但当前所有单据均是「提交即完成」模式——用户在页面点击创建单据后，资产状态立即变更，业务操作一步到位。

在国企固定资产管理实际场景中，这一模式不符合以下业务要求：

- 领用资产需要部门负责人审批确认，不是员工自行领取即生效。
- 跨部门调拨需要调入和调出方双方确认。
- 维修申请需要资产管理员核实故障情况后再安排维修。
- 报废申请必须经过资产管理员和财务人员两级审批，确保资产残值处理合规。
- 所有审批记录需要完整留痕，供后续审计追溯。

因此，本阶段的目标是：**将「提交即完成」改造为「提交申请 → 审批 → 通过后自动执行状态流转」的审批模式。**

## 2. 目标

- 为领用、调拨、维修、报废四个业务类型引入审批流程。
- 审批通过后自动触发第二阶段已有的状态流转逻辑。
- 审批驳回后资产状态不变，单据保留供修改后重新提交。
- 审批过程完整写入操作日志，可追溯。
- 新增我的待办、我的已办、审批历史等页面。

## 3. 审批范围分析：入库是否纳入审批

**结论：入库暂不纳入审批。**

理由如下：
1. 入库的操作主体通常是资产管理员本人。资产管理员在系统中新增资产时，本人即是资产管理的执行角色，不存在「申请人≠执行人」的场景，无需跨角色审批。
2. 入库的实际业务含义是「资产信息录入系统」，资产本身尚未进入使用环节，不涉及部门利益或资源分配，审批的必要性低。
3. 如果后续需要审批（如采购入库需要采购部门确认），可以在第五阶段或后续扩展中加回。

### 本阶段审批范围

| 业务类型 | 是否审批 | 原因 |
|---|---|---|
| 领用 | ✅ 审批 | 涉及资产使用权的转移，需要部门负责人确认 |
| 调拨 | ✅ 审批 | 跨部门资产转移，需要双方确认 |
| 维修 | ✅ 审批 | 需要资产管理员核实故障情况 |
| 报废 | ✅ 审批 | 涉及资产价值核销，需要资产管理员+财务双重确认 |
| 入库 | ❌ 不审批 | 资产管理员直接操作，无需跨角色审批 |

## 4. 用户角色与审批权限

| 角色 | 部门 | 审批范围 |
|---|---|---|
| 普通员工 | 任意部门 | 发起领用/维修申请，查看自己的申请和审批进度 |
| 部门负责人 | 各部门 | 审批本部门员工的领用和维修申请 |
| 资产管理员 | 资产管理部 | 审批调拨、维修、报废申请；查看全部生命周期单据 |
| 财务人员 | 财务部 | 审批涉及固定资产价值核销的报废申请 |
| 审计人员 | 审计部 | 只读查看审批记录和操作日志 |
| 系统管理员 | 信息中心 | 管理审批配置（审批模板、审批节点配置） |

## 5. 功能范围

### 本阶段做

- 审批配置：预定义审批模板和审批节点（领用/调拨/维修/报废各一套）
- 提交审批：用户创建生命周期单据后，可以提交审批
- 我的待办：当前登录用户待审批的单据列表（分页）
- 我的已办：当前登录用户已处理的审批记录（分页）
- 审批通过：审批人同意申请，触发后续状态流转
- 审批驳回：审批人拒绝申请，单据退回，资产状态不变
- 审批历史：查看特定单据的完整审批记录
- 生命周期页面适配：对五个已经实现的生命周期页面进行最小改造
- 审批记录写入 `asset_operation_log`

### 本阶段不做

- 复杂工作流引擎（BPMN / Activiti / Flowable）
- 流程图可视化编辑器
- 多级动态审批配置（如「金额>10万自动加签财务」等动态规则）
- 会签 / 或签 / 转审 / 加签
- 移动端审批
- 企业微信 / 钉钉 / 飞书集成
- 消息通知（短信、邮件、站内信）
- 审批超时自动处理
- 审批委托（他人代审）
- 审批模板的可视化配置界面

## 6. 单据状态

本阶段引入完整的单据状态枚举，覆盖从创建到完成的全生命周期：

| 状态 | 说明 | 所属阶段 |
|---|---|---|
| DRAFT | 草稿 | 第二阶段已存在（维修单） |
| SUBMITTED | 已提交，待部门负责人审批 | 第三阶段新增 |
| APPROVING | 审批中（多级审批时中间状态） | 第三阶段新增 |
| APPROVED | 审批通过，等待执行状态流转 | 第三阶段新增 |
| REJECTED | 审批驳回，退回申请人修改 | 第三阶段新增 |
| COMPLETED | 已完成，状态流转已执行 | 第二阶段已存在 |
| CANCELLED | 已取消 | 第二阶段已存在 |

## 7. 审批规则

### 7.1 基本规则

| 规则 | 说明 |
|---|---|
| 只有 DRAFT 或 REJECTED 状态的单据可以提交审批 | 已提交或审批中的单据不能重复提交 |
| SUBMITTED / APPROVING 状态不能再次提交 | 前端按钮置灰 |
| APPROVED / COMPLETED 单据不能再次审批 | 审批历史已封存 |
| REJECTED 单据可以修改后重新提交 | 修改后状态回到 DRAFT，重新走审批流程 |
| 审批通过后自动执行状态流转 | 复用第二阶段 LifecycleService 中的状态流转逻辑 |
| 审批驳回后资产状态不变 | 单据保留，资产不受影响 |
| CANCELLED 单据只能由申请人自己取消 | DRAFT 状态可取消 |

### 7.2 审批流程规则

每个业务类型的审批节点总数不超过 2 级：

| 业务类型 | 第一级审批 | 第二级审批 |
|---|---|---|
| 领用 | 部门负责人 | —（单级审批） |
| 调拨 | 资产管理员 | —（单级审批） |
| 维修 | 资产管理员 | —（单级审批） |
| 报废 | 资产管理员 | 财务人员 |

报废需要两级审批的原因：报废涉及资产价值核销和残值处理，需要资产管理员确认资产实物状态，财务人员确认账面价值已全部折旧或残值已回收。

### 7.3 审批流转规则

```
申请人创建单据（DRAFT）
        │
        ▼ 申请人点击「提交审批」
     SUBMITTED
        │
        ▼ 审批人打开待办
     APPROVING
        │
        ├── 审批通过 ──► APPROVED ──► (自动执行状态流转) ──► COMPLETED
        │
        └── 审批驳回 ──► REJECTED ──► (申请人修改后重新提交) ──► DRAFT

DRAFT ──► CANCELLED（申请人取消）
```

## 8. 典型流程

### 8.1 领用审批流程

**前置条件：**
- 申请人已登录系统
- 系统中存在 IDLE 状态资产
- 申请人属于某个部门

**主流程：**
1. 申请人进入「资产领用」页面，选择资产，填写领用人、领用部门、领用日期。
2. 申请人点击「提交审批」按钮，单据状态变为 SUBMITTED。
3. 部门负责人登录系统，进入「我的待办」页面，看到该领用申请。
4. 部门负责人点击「审批」，查看领用详情。
5. 部门负责人选择「通过」并填写审批意见。
6. 系统执行状态流转：asset.status = IN_USE，asset.department = 领用部门，asset.keeper = 领用人。
7. 创建领用单（COMPLETED），写入 `asset_operation_log`。

**异常流程：**
- 部门负责人选择「驳回」，填写驳回意见。单据状态变为 REJECTED，资产状态不变。
- 申请人修改领用信息后重新提交审批。

**后置条件：**
- 审批通过：领用单状态 COMPLETED，资产已分配至领用人。
- 审批驳回：单据 REJECTED，资产仍为 IDLE。

### 8.2 调拨审批流程

**前置条件：**
- 调出部门与调入部门不同
- 资产状态为 IDLE 或 IN_USE

**主流程：**
1. 申请人创建调拨单，填写调入部门、调入地点、调入保管人。
2. 提交审批，单据状态变为 SUBMITTED。
3. 资产管理员在待办中看到调拨申请，审批通过。
4. 系统执行调拨流转：更新 asset.department、asset.location、asset.keeper。

**后置条件：**
- 审批通过：调拨单 COMPLETED，资产信息已更新。
- 审批驳回：调拨单 REJECTED，资产信息不变。

### 8.3 维修审批流程

**前置条件：**
- 资产状态为 IDLE 或 IN_USE

**主流程：**
1. 申请人创建维修单，填写故障描述。
2. 提交审批。
3. 资产管理员审批通过。
4. 系统执行：asset.status = REPAIRING，维修单 COMPLETED。
5. 维修完成后，在维修管理页面执行「完成」操作（沿用第二阶段逻辑）。

**后置条件：**
- 审批通过：资产进入维修状态。
- 审批驳回：资产状态不变。

### 8.4 报废审批流程

**前置条件：**
- 资产状态为 IDLE / IN_USE / REPAIRING / WAITING_SCRAP

**主流程：**
1. 申请人创建报废单，填写报废原因、报废日期。
2. 提交审批。
3. 资产管理员审批通过。若为单级资产（如办公设备），审批完成即可执行。
4. 若需要财务确认（如高价值资产），流转至财务人员待办。
5. 财务人员审批通过。
6. 系统执行：asset.status = SCRAPPED，报废单 COMPLETED。

**后置条件：**
- 审批通过：资产报废。
- 审批驳回：资产状态不变。

## 9. 数据规则

| 字段 | 规则 |
|---|---|
| approval_flow.flow_code | 唯一，格式如 APPROVE_RECEIVE |
| approval_flow.business_type | 枚举：RECEIVE / TRANSFER / REPAIR / SCRAP |
| approval_node.approver_role | 枚举：DEPT_HEAD / ASSET_MANAGER / FINANCE |
| approval_record.action | 枚举：APPROVED / REJECTED |
| approval_instance.status | 同单据状态 |

## 10. 异常规则

| 场景 | 处理方式 |
|---|---|
| 提交审批时单据状态不是 DRAFT 或 REJECTED | 返回 400「当前状态不允许提交审批」 |
| 审批时审批实例状态不是 APPROVING | 返回 400「审批已处理或已关闭」 |
| 非审批人操作 | 返回 403「无权操作该审批」 |
| 审批记录写入失败 | 事务回滚，防止审批成功但记录丢失 |

## 13. 验收标准

| 编号 | 验收项 | 验证方式 |
|---|---|---|
| AC-01 | 领用提交审批后状态变为 SUBMITTED | 调用 submit 接口验证 |
| AC-02 | 部门负责人可以通过待办查看到本部门领用申请 | 查看待办列表 |
| AC-03 | 部门负责人审批通过后自动执行领用流转 | 验证资产状态/department/keeper 变化 |
| AC-04 | 部门负责人审批驳回后资产状态不变 | 验证资产状态未变化 |
| AC-05 | 调拨提交审批后经资产管理员审批通过 | 验证调拨后资产信息更新 |
| AC-06 | 维修审批通过后资产状态变为 REPAIRING | 验证资产状态 |
| AC-07 | 报废需两级审批（资产管理员+财务） | 验证两次审批后才能报废 |
| AC-08 | 审批记录可查询 | 调用 records 接口 |
| AC-09 | 审批操作写入 asset_operation_log | 验证日志表有对应记录 |
| AC-10 | DRAFT 单据不能直接完成业务流转 | 验证创建后资产状态不变 |
| AC-11 | REJECTED 单据修改后可重新提交 | 验证可以重新走审批 |
| AC-12 | APPROVED 单据不能再次审批 | 验证返回 400 |
| AC-13 | 后端构建通过 | mvn -DskipTests package |
