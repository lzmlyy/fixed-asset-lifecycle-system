## 四、审批流专题测试

第三阶段审批流涵盖 **4 种业务类型**：

| 业务类型 | 审批级别 | 审批角色 |
|---|---|---|
| 领用 (RECEIVE) | 1 级 | 资产管理员 |
| 调拨 (TRANSFER) | 1 级 | 资产管理员 |
| 维修 (REPAIR) | 1 级 | 资产管理员 |
| 报废 (SCRAP) | **2 级** | 资产管理员 → 财务人员 |

### 4.1 核心业务流程

```
创建单据(DRAFT)
     │
     ▼ 用户提交审批 (POST /api/approval/submit)
  审批中 (APPROVING)
     │
     ├── 审批通过 → (还有下一节点?)
     │       ├── 是 → 推进到下一节点，继续审批
     │       └── 否 (最终节点) → 执行状态流转 → 单据完成(COMPLETED)
     │
     └── 审批驳回 → 单据退回(REJECTED)，资产状态不变
```

### 4.2 调拨流程图解

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 创建调拨单 │───▶│ 提交审批  │───▶│ 审批通过  │───▶│ 执行流转  │
│ (DRAFT)   │    │          │    │ (1级)    │    │          │
│ asset不变  │    │ APPROVING│    │          │    │ dept/    │
└──────────┘    └──────────┘    └──────────┘    │ loc/     │
                                                │ keeper   │
                                                │ 更新     │
                                                └──────────┘
```

### 4.3 报废两级审批图解

```
┌──────────┐    ┌─────────────┐    ┌──────────────┐    ┌──────────┐
│ 创建报废单 │───▶│ 一级审批通过  │───▶│ 二级审批通过   │───▶│ 执行流转  │
│ (DRAFT)   │    │ 资产管理员   │    │ 财务人员      │    │          │
│ asset不变  │    │ asset仍然   │    │ asset→       │    │ SCRAPPED │
└──────────┘    │ 不变        │    │ SCRAPPED     │    │          │
                └─────────────┘    └──────────────┘    └──────────┘
```

### 4.4 审批流完整链路验证结果

| 测试场景 | 步骤 | 预期结果 | 实际结果 |
|---|---|---|---|
| 领用审批 | 创建领用单(DRAFT) → 提交审批 → 审批通过 | asset.status = IN_USE, department/keeper 更新 | ✅ 通过 |
| 调拨审批 | 创建调拨单(DRAFT) → 提交审批 → 审批通过 | 部门/地点/保管人更新 | ✅ 通过 |
| 维修审批 | 创建维修单(DRAFT) → 审批通过 → 维修完成 | REPAIRING → IN_USE | ✅ 通过 |
| 报废·一级 | 创建报废单(DRAFT) → 一级审批通过 | asset 状态不变 | ✅ 通过 |
| 报废·二级 | 二级审批通过 | asset.status = SCRAPPED | ✅ 通过 |
| 审批驳回 | 审批人驳回申请 | asset 状态不变，INSTANCE=REJECTED | ✅ 通过 |
| 重复提交 | 已提交审批的单据再次提交 | 返回 400 业务错误 | ✅ 通过 |
| 非法状态 | 已完成审批再 approve | 返回 400 | ✅ 通过 |
| 不存在实例 | GET/POST 不存在的审批实例 | 返回 HTTP 404 | ✅ 通过 |

### 4.5 审批通过的资产状态流转

```
审批业务类型       资产原状态       审批通过后状态
────────────      ─────────       ────────────
领用 (RECEIVE)    IDLE            IN_USE (+ 部门/保管人更新)
调拨 (TRANSFER)   IDLE/IN_USE     IN_USE (+ 部门/地点/保管人更新)
维修 (REPAIR)     IDLE/IN_USE     REPAIRING
报废 (SCRAP)      —               SCRAPPED
入库 (INBOUND)    —               IDLE (无需审批)
```

## 六、发现的问题与修复建议

本次测试共发现 **2 个微小的可优化项**，均 **不影响** 现有功能的正常使用。

### 🟡 #1 requireAsset() 不存在资产返回 400 而非 404

- **问题**：`LifecycleService.requireAsset()` 方法在资产不存在时使用 `ResultCode.BAD_REQUEST`（HTTP 400），而非 `ResultCode.NOT_FOUND`（HTTP 404）
- **建议**：将状态码统一为 404 以符合 REST 语义
- **修复工作量**：修改 1 行代码

### 🟡 #2 AssetUpdateRequest 字段全部为必填

- **问题**：编辑资产时所有字段均为 `@NotNull`，导致不能只传需要修改的字段进行部分更新
- **建议**：在 Service 层做差异更新，允许只传需要修改的字段
- **修复工作量**：中等（修改 DTO 和 Service 层逻辑）
